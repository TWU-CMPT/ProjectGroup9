Guidelines

1. Technical Guidelines

       The app development will be done using the Xcode(version 7.0.1)

       The programming language will be Swift.

       The database management will be done using SQLite and the app is able to read and collect data from the database

       Main Frameworks we could possibly use:

o   Accounts can use this to grant user access without the need to type their username or password. Otherwise, the user can create and save an account from within the app

o   AVFoundation The heart rate detection will be done using the camera and some classes this API

o   Core Graphics main framework to create the visuals in the app

o   Swift standard library

2. Ethical and Legal Issue

       'Breathify' is not a medical app. It should not be used for medical treatment and it is designed for use by nonclinical population.

       Only public APIs will be used, as specified in the App Store Guidelines 2.5.1. https://developer.apple.com/app-store/review/guidelines/#software-requirements

 

 

3. Privacy Policy

'Breathify' will respect the privacy of its users and provide a safe and secure environment. Users will have options to control or restrict the use of their information, such as the availability of the results obtained from the exercises in the app. This section will specify details on Breathify's privacy policy.

What information will Breathify collect:

       Personal information such as name/username, age, gender and email.

       Breathify account information such as account number/password

 

How the information will be used internally:

       Name/username, age, and gender information will be used to personalize the user's profile

When and to whom will the information be shared with:

       With permission of the user, their log data and scores will be uploaded onto the server. Friends of the user who are also using the app could see the user's progress and scores.

       The user's information can be accessed whenever the user's friends request to view their profile with internet connection.

Where the information will be stored:

       If the user chooses to share their results, it will be uploaded onto the server. If the user wishes to opt out, their results and scores will stay locally saved onto their phone.

How secure the information is:

       The user must be aware that while the team will do their best to maintain the security of personal information, we are unable to fully guarantee security from all possible risks and breaches.

Third party services:

       The app will have an option to authenticate with Facebook/Google logins. This is for the purpose to obtain basic information for identification. Personal data, log data or scores will not be shared directly.

Data Requirements

Summary of I/Os:

       File input and database format

       The main database (.DB3) file of SOLite

  user info (name/age/gender in View Profile)

  exercise results(game/timer based score, heartrate result in Exercise Library)

       Heart rate input from the user

       touch the camera and hold

       Keyboard input

       Login (account/password) or use Facebook/Google to authenticate login credentials

       Edit user profile: set name/user age/gender

       Recovering a password (new password, old password)

       Feedback menu (Typing Feedback)

       Resources menu (Typing emails)

       View profile menu (add/change profile, setting)

       Exercise menu (Typing in filter setting, typing comment(feedback))

       Button click

       Open the app, profile selection (create new account/online user select), login confirmed, forgot password, recovery confirmed

       Main menu options selections (Give feedback, view profile, view exercise, view resources)

       Feedback submit, view profile selection (setting, back), setting selection (submit, back, etc.), resources selection (show info, show resource library, show contact list, select article, select contact, select email contact, dial contact, call end, back in article, back in contact list/info, back in resource library, back in resource menu)

        Exercise selection(show filter setting, apply in filter setting, cancel in filter setting, timer selected, game selected, interact in both game/timer, pause, quit/resume/restart in pause menu, show result in score screen, replay/back/give to main menu/give feedback in result screen, back/submit in feedback)

       Close the app

       Output:

       Login display (login prompt, profile creation form, password recovery form)

       Main menu display (menu options), Feedback display(form), view profile display(user info, account and app setting)

       Exercise display (filter setting, pause menu, main game(shapes, motions), score, results, heart rate graph/result, feedback/comment)

       Resources display (app info, contact info, articles/information, email form)

 

 

Feature Priority

Version 1

       The app should have one exercise playable in both timer and game based modes.

       The heart rate monitor should be able to track and record heart rates accurately.

       The timer based exercise trainer should be functioning properly.

       The game based exercise trainer should be functioning properly.

       The results screen should display the exercise, duration, and heart rate data.

       Both modes should be able to be paused and restarted through the pause menu.

       Both modes should be able to be muted and unmuted through the pause menu.

       Both modes should be able to quit to the Exercise Library menu through the pause menu.

       Scoring should be functional for the game based exercise trainer.

       The app information and About Us page should be created and accessible.

Version 2

       The databases for user profiles and exercise records should be created and populated.

       The API for accessing the databases should be setup and available.

       Users should be able to create local and online user profiles.

       Online users should be able to reset their password.

       Authentication for online user login should be complete.

       Navigation UI elements should be complete for all but the View Resources section.

       The filter for the exercises in the Exercise Library list should be functional.

       Users should be able to view their own profile with all their data displayed.

       Users should be able to view their exercise history.

       Users should be able to access and change their account information and settings.

       Users should be able to change application settings.

       Users should be able to add and remove friends from their friends list.

       Users should be able to view the profiles of people on their friends list.

       Users should be able to view the exercise history of the people on their friends list (if they opted-in to sharing their usage data).

Version 3

       Resource articles should be written, sourced, and accessible.

       View Resources section menus should be complete.

       Contacts should be compiled and accessible through the View Contacts section.

       Users should be able to leave a rating and feedback for the exercises they complete.

       Users should have access to tutorials that teach how to use both modes of breathing exercises.

       Users should be able to set reminders for exercises of their choice.

       Users should be able to create profiles using a social media login method (specifically Google and Facebook).

       Users should be able to create their own breathing patterns to add to their exercise library.
